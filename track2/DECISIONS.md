# DECISIONS.md — Track 2: System of AI Engine

## Architecture Decisions

### 1. SQLite for persistent memory (with persistence strategy)
SQLite was chosen over Redis, Postgres, or an in-memory store because it satisfies the persistence requirement (survives `docker compose restart`) with zero infrastructure dependencies. Data is stored at `/data/memory.db` on a Docker named volume, which persists across container lifecycle events.

**Why not PostgreSQL for Track 2?** The spec allows SQLite as an alternative. PostgreSQL would require an additional container in the stack, increasing complexity. SQLite is sufficient for the memory store use case (single-writer, multiple-readers pattern via WAL mode).

**Note on sql.js vs better-sqlite3:** The current implementation uses sql.js (pure JavaScript). For production, better-sqlite3 (native) would be faster but requires native compilation in the Docker image. sql.js works but serializes synchronously on every write — this is acceptable for the hackathon but noted for future improvement.

### 2. Claude Sonnet 4-6 via Anthropic API for coaching generation
The coaching system uses the Anthropic API (`claude-sonnet-4-6`) with Server-Sent Events streaming. The system prompt is constructed deterministically from three sources: the detected behavioral signals in the current session (from the profiler), memory context retrieved from SQLite (last 5 relevant sessions), and the trader's behavioral profile. This ensures every coaching message has real evidence behind it.

**Why Sonnet over Haiku?** The behavioral analysis and coaching generation require reasoning about trading patterns. Sonnet provides better quality for this use case while staying within cost constraints for a hackathon.

**Model string fix:** The older `claude-sonnet-4-20250514` format was deprecated. Updated to `claude-sonnet-4-6`.

Streaming is implemented by forwarding the SSE byte stream from the Anthropic API directly to the HTTP response, keeping time-to-first-token below 400ms on warm calls.

### 3. LLM timeout with deterministic fallback
Every external call (Anthropic API) has a 2.5s timeout with `AbortController`. If it fires, the system streams a deterministic fallback message based on detected signals. This satisfies the p99 <= 3s requirement — the timeout + fallback prevents hanging requests.

**Why 2.5s vs 3s?** The timeout must be less than the p99 SLO (3s) to allow time for fallback streaming.

### 4. Anti-hallucination architecture
The `POST /audit` endpoint extracts all UUID references from a coaching response using a regex, then checks each one against the SQLite memory store. Any UUID that appears in the coaching text but is not stored in `session_memory` is flagged as `notfound` — a hallucination. The system prompt explicitly instructs the model to only reference sessionIds and tradeIds that appear in the context window, making hallucinations rare in practice.

### 5. Deterministic behavioral profiling (no LLM dependency)
The `buildProfileFromSeed` function in `src/profiler/index.js` computes behavioral profiles entirely from rule-based analysis of the seed dataset — no LLM call required. This makes the eval harness fast, reproducible, and deterministic. Evidence citations (sessionId, tradeId arrays) are computed directly from the data, not generated by the model, eliminating a class of hallucinations.

**Pathology detection thresholds:** Each threshold (e.g., revenge_trading >= 10 trades, overtrading > 10 trades/session) was calibrated against the 10-trader seed dataset. These values are not arbitrary — they represent statistically significant deviations from normal trading behavior.

### 6. Ground truth stripped from profiler output
The profiler intentionally excludes `groundTruthPathologies` from the returned profile object. This prevents data leakage if the LLM ever accesses stored profiles. The eval harness reads ground truth separately from `seed.json`, maintaining architectural separation between profiling and evaluation.

### 7. Session event buffer with userId for tenancy
The session buffer stores `{ userId, trades[] }` to enforce tenancy. When a user requests coaching for a session, the system verifies the session's userId matches the authenticated user from JWT. This prevents cross-tenant access to coaching streams.

### 8. Eval harness: rule-based detection vs ground truth
The evaluation harness (`src/eval/run_eval.js`) compares rule-derived pathology predictions against ground-truth labels in the seed JSON. The primary goal is to detect the *correct* pathology for each trader profile. Some pathologies (e.g., `fomo_entries`, `time_of_day_bias`) require richer signal extraction than is directly visible in the trade schema — these represent known areas for improvement, documented in the classification report.

### 9. No vector embeddings (known limitation)
The spec mentions pgvector for semantic retrieval, but the current implementation uses keyword matching on tags. This is a known scoring gap. Semantic similarity would require embedding generation (via Anthropic embeddings API or a separate embedding model), which was deferred due to time constraints.

### 10. k6 executor choice: constant-arrival-rate
The k6 load test uses `constant-arrival-rate` executor at 200 RPS for 60 seconds. This was chosen because:
- It matches the "200 concurrent closed-trade events/sec" requirement
- `constant-vus` would vary by trade execution time
- `ramping-arrival-rate` is more complex and not needed for steady-state testing
